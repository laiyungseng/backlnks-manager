# Changelog - v1.5.5

## [2026-03-07] - Domain Manager Fix and Completed Timestamp

### Added
- Added `completed_at` to the `projects` table when a project transitions to a "Completed" status.
- Added migration script command to `migration.sql` to append the `completed_at` column.

### Fixed
- Fixed bug in the `VendorManager` and `DomainManager` components where deleting a row caused a `Runtime TypeError: 0 is not iterable` due to Glide Data Grid v6 API differences in `CompactSelection.toArray()` behavior.
- Fixed a schema cache error when finalizing a project by updating `project_details` JSON object instead of a direct `status` column on `projects` table in `placementProcessor.js`.
- Fixed an issue where the `completed_at` timestamp was not triggered in the database despite the UI showing 'Completed'. The vendor `saveVendorProgress` logic now cross-references the actual target order quantity instead of the virtual row count to accurately trigger status completion.
- Ensured that `domains.created_at` logic inside `placementProcessor.js` only triggers upon the initial domain creation and does not overwrite or update on subsequent domain syncs.
- Fixed an issue where the database status failed to explicitly map back to `complete_date` when completed. Replaced incorrect `completed_at` payload insertions with `complete_date` in the vendor actions.
- Resolved a data parsing error on the Vendor InProgress and Completed dashboards (`projects.project_name does not exist`) by refactoring the Supabase fetch queries to correctly target the `project_details` JSONB array per the current schema standards.
- Fixed a vendor form submission error caused by pushing `complete_date` to `projects_hub`. The virtual `projects_hub` table now safely uses `completed_at` to evaluate fulfillment, while the master `projects` log records mapping correctly against `complete_date`.
- Fixed the sorting schema on vendor pages to accurately query `projects.created_date` instead of the deprecated `projects.created_at`.
